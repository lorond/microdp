# Transaction Emulator

Небольшой сервис фоновой генерации демо-нагрузки.

## Назначение

- Периодически создает транзакции через wallet API.
- Периодически отправляет clickstream-события через clickstream API.
- Дает поток данных для CDC, Kafka и витрин без ручного UI-сценария.

## Требования

- Не пиши напрямую в PostgreSQL или Kafka; эмулятор должен использовать публичные HTTP API.
- Активности запускаются параллельно через `ThreadPoolExecutor`. Каждая активность = одна короткоживущая сессия одного пользователя со стримом событий в реальном времени (`route_change → page_enter → mouse_move/click → page_leave` с реальными sleep'ами между фазами) и опциональной транзакцией в конце.
- `TRANSACTIONS_PER_MINUTE` — целевой темп запуска новых активностей (по умолчанию 30/мин, проброшена через `.env`). Фактический темп завершённых активностей сверху ограничен `MAX_CONCURRENT_ACTIVITIES` и средней длительностью активности.
- `MAX_CONCURRENT_ACTIVITIES` (по умолчанию 4) — потолок параллельных активностей. В любой момент времени один user может быть максимум в одном слоте: scheduler выбирает только из user'ов, у которых нет in-flight активности, чтобы пользователи реально работали параллельно, а не дублировались в нескольких слотах.
- Пул пользователей эмулятор берёт через `GET /api/users` у wallet-api при старте (с ретраями, пока Postgres init-скрипты не доготовят таблицу). Жёстко прописан только `DEMO_USER_ID` (по умолчанию `00000000-0000-0000-0000-000000000001`) — этого пользователя эмулятор НЕ трогает, чтобы баланс под ручную демонстрацию в UI не «съезжал». Остальные seeded'ятся `infra/postgres/init/002_seed_demo_users.sh` со случайными UUID, поэтому никаких других хардкод-ID в коде эмулятора, .env или compose быть не должно.
- Полностью отключается переменной `EMULATOR_ENABLED=false` — контейнер стартует и сразу завершается; используется для воспроизводимых демо без фонового трафика.
- По умолчанию нагрузка должна оставаться небольшой и подходящей для локального стенда на 20 GB RAM.
- SIGTERM/SIGINT ставит общий `stop_event`; in-flight активности проверяют его между фазами и выходят, scheduler ждёт их через `pool.shutdown(wait=True)`.
- Ошибки активности логируй и продолжай работу scheduler'а, чтобы временная недоступность сервисов не останавливала контейнер.
- Не добавляй сложный сценарный движок без явной задачи.

## Проверки

- `python3 -m py_compile apps/transaction-emulator/emulator.py`
- `docker compose -f infra/docker-compose.yml --env-file .env.example build transaction-emulator`
